# 核心模块详细设计

## 一、数值养成体系

### 1.1 属性体系

#### 一级属性 (基础属性)
| 属性名 | 说明 | 计算方式 |
|--------|------|----------|
| 生命上限 (hp) | 武将最大生命值 | 基础 + 成长*等级 + 装备 + 增益 |
| 攻击力 (attack) | 物理/法术攻击 | 基础 + 成长*等级 + 装备 + 增益 |
| 防御力 (defense) | 物理/法术防御 | 基础 + 成长*等级 + 装备 + 增益 |
| 速度 (speed) | 行动顺序 | 基础 + 成长*等级 + 装备 + 增益 |

#### 二级属性 (战斗属性)
| 属性名 | 说明 | 来源 |
|--------|------|------|
| 暴击率 (critRate) | 暴击概率 | 天赋/装备/符文 |
| 暴击伤害 (critDamage) | 暴击伤害倍率 | 天赋/装备/符文 (默认150%) |
| 命中率 (hitRate) | 命中概率 | 基础100% - 闪避率 |
| 闪避率 (dodgeRate) | 闪避概率 | 天赋/装备/符文 |
| 效果命中 (effectHit) | 控制技能命中率 | 天赋/装备/符文 |
| 效果抵抗 (effectResist) | 抵抗控制概率 | 天赋/装备/符文 |
| 伤害加成 (damageAdd) | 伤害增加百分比 | 羁绊/天赋/装备 |
| 伤害减免 (damageReduce) | 受到伤害减少 | 羁绊/天赋/装备 |

### 1.2 属性计算公式

#### 标准计算模型
```
最终属性值 = (基础值 + 固定加成1 + 固定加成2 + ...) 
           × (1 + 百分比加成1) 
           × (1 + 百分比加成2) 
           × ...
```

#### 各养成维度属性计算顺序

```
1. 基础属性 (base)
   └─ 武将初始属性 (来自配置表)

2. 等级加成 (level)
   └─ 成长值 × 等级 (固定加成)

3. 升星加成 (star)
   ├─ 基础属性百分比加成
   └─ 成长值百分比加成

4. 突破加成 (quality)
   └─ 基础属性百分比加成

5. 天赋加成 (talent)
   ├─ 固定值加成
   └─ 百分比加成

6. 羁绊加成 (bond)
   └─ 百分比加成 (所有羁绊同时生效)

7. 装备加成 (equipment)
   ├─ 装备基础属性 (固定值)
   ├─ 装备强化属性 (固定值)
   ├─ 装备主属性 (百分比/固定)
   ├─ 装备副属性 (百分比/固定)
   └─ 套装效果 (百分比/固定)

8. 符文加成 (rune)
   ├─ 符文基础属性
   ├─ 符文强化属性
   └─ 符文套装效果

9. 城池科技加成 (tech)
   └─ 全局百分比加成 (所有武将)
```

### 1.3 战力计算

```typescript
// 战力权重配置
const POWER_WEIGHTS = {
  hp: 0.5,           // 每1点生命加0.5战力
  attack: 2.0,       // 每1点攻击加2战力
  defense: 1.5,      // 每1点防御加1.5战力
  speed: 1.0,        // 每1点速度加1战力
  critRate: 100,     // 每1%暴击率加100战力
  critDamage: 50,    // 每1%暴击伤害加50战力
  damageAdd: 80,     // 每1%伤害加成加80战力
  damageReduce: 80,  // 每1%伤害减免加80战力
  effectHit: 60,     // 每1%效果命中加60战力
  effectResist: 60,  // 每1%效果抵抗加60战力
};

// 武将战力 = Σ(属性值 × 属性权重) + 技能战力 + 装备战力
// 总战力 = Σ武将战力 + 羁绊加成战力
```

### 1.4 数值校验机制

1. **最大值限制**: 每个属性设置上限，防止溢出
2. **合理性校验**: 属性值与等级、星级匹配校验
3. **组合校验**: 特定属性组合的上限校验
4. **战力校验**: 战力变化超过阈值告警

## 二、战斗系统

### 2.1 战斗流程

```
┌─────────────┐
│  战斗初始化  │  加载阵容、计算属性、初始化状态
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  回合开始   │  速度排序，确定行动顺序
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  行动阶段   │  当前最快单位行动
│  (循环)     │  - 选择技能
│             │  - 选择目标
│             │  - 技能结算
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  回合结束   │  状态结算 (中毒/灼烧等)
└──────┬──────┘
       │
       ├─ 检查胜负 ── 胜/负 ──► 战斗结束
       │
       └─ 未分胜负 ──► 下一回合
```

### 2.2 技能系统

#### 技能类型
| 类型 | 说明 | 冷却 |
|------|------|------|
| 普通攻击 | 无消耗，默认技能 | 0回合 |
| 主动技能 | 需要手动/自动释放 | 有冷却 |
| 被动技能 | 常驻效果，无需释放 | 无 |
| 觉醒技能 | 满足条件自动触发 | 长冷却 |

#### 技能效果类型
```
伤害类:
  - 物理伤害 (受物理防御影响)
  - 法术伤害 (受法术防御影响)
  - 真实伤害 (无视防御)
  - 持续伤害 (中毒/灼烧/流血)

治疗类:
  - 单体治疗
  - 群体治疗
  - 持续治疗 (回血buff)

增益类:
  - 属性提升 (攻击/防御/速度等)
  - 护盾
  - 无敌
  - 复活

减益类:
  - 属性降低
  - 持续伤害
  - 控制类 (眩晕/沉默/冰冻/石化/嘲讽)
```

#### 技能结算优先级
```
1. 被动技能触发 (入场触发的被动)
2. 技能命中判定
3. 暴击判定
4. 伤害计算
5. 护盾吸收
6. 最终伤害结算
7. 状态效果添加
8. 击杀后的被动触发
```

### 2.3 伤害计算公式

```
基础伤害 = 攻击力 × 技能倍率 × (1 + 伤害加成总和)

防御减伤 = 防御 / (防御 + 600)  (600为防御系数，可配置)

最终伤害 = 基础伤害 × (1 - 防御减伤) 
         × (1 - 伤害减免总和) 
         × 暴击倍率(如果暴击) 
         × 克制系数(职业/阵营克制) 
         × 随机波动(0.95~1.05)
```

### 2.4 战报系统

#### 战报数据结构
```typescript
interface BattleReport {
  reportId: string;
  battleType: number;
  attacker: BattleSide;
  defender: BattleSide;
  winner: number; // 1攻 2守
  totalRounds: number;
  rounds: RoundData[];
}

interface RoundData {
  round: number;
  actions: ActionData[];
  endEffects: StatusEffect[]; // 回合结束结算
}

interface ActionData {
  actorId: string;      // 行动者ID
  actorSide: number;    // 1攻 2守
  skillId: number;      // 技能ID
  skillName: string;
  targets: TargetData[];
  isCrit?: boolean;
}

interface TargetData {
  targetId: string;
  damage?: number;      // 伤害 (负数为治疗)
  hpChange: number;     // 实际血量变化
  isCrit?: boolean;     // 是否暴击
  isDodge?: boolean;    // 是否闪避
  effects?: StatusEffect[]; // 附加状态
}

interface StatusEffect {
  effectType: string;   // 状态类型
  value: number;        // 数值
  duration: number;     // 剩余回合
  fromSkill?: number;   // 来源技能
}
```

#### 战报压缩策略
1. **差异存储**: 只存储血量变化，不存绝对血量
2. **增量编码**: 状态变化用位运算存储
3. **Gzip压缩**: 存储前Gzip压缩
4. **冷热分离**: 近期战报存热库，历史战报存冷库

### 3.2 积分规则

#### 基础积分
```
胜利: +20~30分 (根据双方实力差计算)
失败: -15~25分
平局: ±0分

连胜加成:
  3连胜: +3分
  5连胜: +5分
  10连胜: +10分
  20连胜以上: +15分
```

#### ELO匹配算法
```
预期胜率: Ea = 1 / (1 + 10^((Rb - Ra) / 400))
积分变化: ΔR = K × (Sa - Ea)

其中:
  Ra, Rb: 双方当前积分
  Ea: A的预期胜率
  Sa: 实际结果 (1胜 0.5平 0负)
  K: 系数 (新人40，普通32，高端局20)
```

### 3.4 跨服联赛设计

#### 赛程阶段
1. **报名期**: 3天，达到指定段位可报名
2. **小组赛**: 5天，每组8人，循环对战
3. **淘汰赛**: 3天，单败淘汰，BO3制
4. **决赛周**: 半决赛、决赛，BO5制

#### 积分规则
- 小组赛: 胜3分，负0分
- 淘汰赛: 晋级制，胜者晋级
- 每场战斗有独立积分用于同分排名

#### 奖励
- 参与奖: 所有报名玩家
- 排名奖: 按最终排名发放
- 王者奖励: 冠军特殊称号/头像框

## 四、城池经营系统

### 4.1 建筑体系

| 建筑 | 功能 | 效果 |
|------|------|------|
| 主殿 | 核心建筑 | 决定其他建筑等级上限 |
| 农田 | 产粮 | 产出粮食资源 |
| 银库 | 产银 | 产出银币资源 |
| 兵营 | 守军 | 提升守城部队数量/战力 |
| 校场 | 经验 | 提升武将经验获取速度 |
| 铁匠铺 | 装备 | 提升装备强化成功率/解锁高级锻造 |
| 仓库 | 存储 | 提升资源存储上限 |
| 城墙 | 城防 | 提升城池防御力 |

### 4.2 资源产出机制

#### 产出公式
```
每秒产出 = 基础产出 × (1 + 建筑等级加成) × (1 + 科技加成) × ...

累计产出 = 每秒产出 × 经过秒数
实际获得 = min(累计产出, 仓库上限 - 当前数量)
```

#### 结算策略
1. **在线结算**: 每分钟结算一次
2. **离线结算**: 玩家上线时一次性结算
3. **定时任务**: 后台定时批量结算 (防止玩家不上线)
4. **防刷机制**:
   - 时间校准: 用服务器时间，不信任客户端时间
   - 产出上限: 最多累计24小时产出
   - 异常检测: 产出速率异常告警

### 4.3 建筑升级

```
升级消耗: 资源数量 = 基础消耗 × (倍率 ^ 等级)
升级时间: 时间 = 基础时间 × (倍率 ^ 等级)

加速机制:
  - 钻石加速: 直接完成
  - 道具加速: 使用加速卷轴
  - 免费加速: 小于5分钟可免费完成
```

## 五、配置中心设计

### 5.1 配置类型

| 配置类型 | 存储格式 | 更新频率 | 示例 |
|----------|----------|----------|------|
| 武将配置 | JSON | 低 | 武将属性、技能 |
| 装备配置 | JSON | 中 | 装备属性、套装 |
| 技能配置 | JSON | 中 | 技能效果、数值 |
| 活动配置 | JSON | 高 | 活动时间、奖励 |
| 数值常量 | JSON | 低 | 战斗公式参数 |
| 段位配置 | JSON | 低 | 段位积分表 |

### 5.2 版本管理

#### 版本状态流转
```
草稿 → 测试 → 灰度 → 正式
 ↑       ↓      ↓      ↓
 └───────┴──────┴──────┘
        (回退/回滚)
```

#### 灰度发布策略
1. **按比例灰度**: 1% → 10% → 50% → 100%
2. **按用户分组灰度**: 白名单用户优先
3. **按渠道灰度**: 指定渠道先更新
4. **AB测试**: 多版本并行，数据分析后决定

#### 回滚机制
- 一键回滚到上一个版本
- 保留最近10个版本的完整备份
- 回滚自动记录操作日志

### 5.3 配置校验

1. **语法校验**: JSON格式校验
2. **数据类型校验**: 字段类型、范围校验
3. **引用校验**: 技能ID、装备ID等引用存在性检查
4. **数值校验**: 数值合理性检查 (上限、比例)
5. **组合校验**: 羁绊、套装等组合关系校验

## 六、跨服架构设计

### 6.1 架构模式

采用**中心服 + 游戏服**的星型架构:

```
┌─────────┐   ┌─────────┐   ┌─────────┐
│ 游戏服1 │   │ 游戏服2 │   │ 游戏服N │
└────┬────┘   └────┬────┘   └────┬────┘
     │             │             │
     └─────────────┼─────────────┘
                   │
            ┌──────▼──────┐
            │  跨服中心服  │
            └──────┬──────┘
                   │
            ┌──────▼──────┐
            │  跨服数据库  │
            └─────────────┘
```

### 6.2 跨服通信方式

1. **HTTP API**: 非实时数据同步
2. **RPC调用**: 游戏服 → 跨服服 实时调用
3. **消息队列**: 异步消息通知
4. **WebSocket**: 战斗实时推送

### 6.3 跨服对战流程

```
1. 玩家在游戏服报名
2. 游戏服将玩家数据同步到跨服服
3. 跨服服匹配对手、创建战斗
4. 战斗在跨服服进行，实时推送进度
5. 战斗结束，结果回传到游戏服
6. 游戏服发放奖励
```

### 6.4 数据同步策略

1. **参战数据同步**: 参赛时全量同步武将、装备数据
2. **增量同步**: 玩家数据变化时，增量更新到跨服服
3. **数据隔离**: 跨服服只保留对战所需数据，不存完整数据
4. **数据清理**: 赛季结束后清理跨服临时数据
